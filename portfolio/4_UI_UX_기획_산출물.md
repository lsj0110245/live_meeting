### 1.2 핵심 플로우 상세

#### Flow A: 실시간 녹음 → 회의록 자동 생성

```
[대시보드]
    │ "실시간 녹음" 버튼 클릭
    ▼
[녹음 페이지]
    │ 1. 메타데이터 입력 (제목, 참석자, 유형)
    │ 2. "녹음 시작" 버튼 클릭
    │    └─ 마이크 권한 요청 → 허용
    │
    │ === 녹음 진행 중 ===
    │ • 실시간 자막이 화면에 표시
    │ • 3분마다 중간 요약 자동 생성
    │ • 녹음 시간 타이머 표시
    │
    │ 3. "녹음 종료" 버튼 클릭
    │    └─ "AI가 회의록을 정리하고 있습니다..." 로딩 표시
    ▼
[회의 상세 페이지]
    • 구조화된 AI 회의록 표시
      (📅 요약 / 📌 안건 / ✅ 결론 / 📝 향후 계획)
    • 전사 텍스트 타임라인 표시
    • 오디오 플레이어 + 시크(Seek) 지원
    • CSV/XLSX 내보내기
```

#### Flow B: 파일 업로드 → 비동기 처리

```
[대시보드]
    │ 파일을 드래그 앤 드롭 (또는 클릭 업로드)
    ▼
[업로드 모달]
    │ 1. 파일 선택 (MP3, WAV, M4A, MP4, WebM 등)
    │ 2. 메타데이터 입력 (제목, 참석자 등)
    │ 3. "업로드" 버튼 클릭
    ▼
[대시보드 — 처리 중]
    │ • 회의 카드에 "처리 중" 배지 표시
    │ • 프로그레스 바 (0% → 100%)
    │ • 실시간 진행률 업데이트 (2초 단위 폴링)
    ▼
[대시보드 — 완료]
    │ • 회의 카드에 "완료" 배지 표시
    │ • 클릭 → 회의 상세 페이지 이동
    ▼
[회의 상세 페이지]
    • AI 회의록 + 전사 타임라인 + 오디오 플레이어
```

#### Flow C: 이어서 녹음

```
[회의 상세 페이지]
    │ "이어서 녹음하기" 버튼 클릭
    ▼
[녹음 페이지 — 이어서 녹음 모드]
    │ • 기존 회의 ID 전달
    │ • 이전 전사 세그먼트 이어받기
    │ • segment_index 연속성 유지
    ▼
[녹음 종료]
    │ • 기존 오디오 + 새 오디오 FFmpeg concat 병합
    │ • Duration 합산
    │ • 전체 내용으로 회의록 재생성
```

## 4. 성능 최적화

### 4.1 프론트엔드 최적화 전략

| 최적화 항목 | 구현 방법 | 효과 |
|------------|----------|------|
| **SSR (서버 사이드 렌더링)** | Jinja2 템플릿 렌더링 | 초기 로딩 속도 향상, SEO 개선 |
| **JS 모듈 분리** | 페이지별 독립 JS 파일 | 불필요한 코드 로딩 방지 |
| **CSS 변수** | `:root` CSS Custom Properties | 일관된 테마, 런타임 변경 가능 |
| **ETag 캐싱** | 오디오 파일 ETag 헤더 | 동일 파일 재요청 시 304 반환 |
| **Range Request** | HTTP 206 Partial Content | 오디오 파일 시크(Seek) 지연 최소화 |
| **진행률 폴링 최적화** | 완료 시 폴링 자동 중지 | 불필요한 API 호출 제거 |

### 4.2 백엔드 성능 전략

| 최적화 항목 | 구현 방법 | 효과 |
|------------|----------|------|
| **비동기 처리** | `asyncio.to_thread()` GPU 연산 분리 | WebSocket 이벤트 루프 차단 방지 |
| **BackgroundTask** | 파일 업로드 STT 비동기 처리 | API 응답 즉시 반환 (202 Accepted) |
| **STT Pre-warm** | WebSocket 연결 시 모델 사전 로딩 | 첫 전사 지연 최소화 |
| **STT 이원화** | 실시간(속도) vs 파일(정확도) | 상황별 최적 UX |
| **SHA-256 중복 방지** | 파일 해시 비교 | 동일 파일 재처리 방지 |
| **Map-Reduce** | 10,000자 초과 시 청킹 요약 | 메모리 초과 없이 긴 회의 처리 |

### 4.3 Lighthouse 성능 개선 포인트 (자체 분석)

| 카테고리 | 현재 예상 | 개선 방안 |
|----------|----------|----------|
| **Performance** | 🟡 75~85 | 이미지 최적화(WebP), JS 번들 최소화, Lazy Loading |

## 5. 기획 결정 및 근거

### 5.1 주요 기획 결정

| 결정 | 배경 | 근거 |
|------|------|------|
| **실시간 + 파일 업로드 이원화** | 사용 시나리오가 다름 | 실시간: 속도 우선 → 사후 정밀 분석 가능. 파일: 정확도 우선 |
| **3분 중간 요약** | 긴 회의 시 최종 요약 품질 저하 | Map-Reduce 사전 분할 → 최종 요약 정확도 향상 |
| **메타데이터 AI 자동 추출** | 사용자 입력 부담 경감 | LLM이 제목, 회의 유형, 참석자를 전사 내용에서 자동 추출 |
| **폴더 기반 분류** | 태그 vs 폴더 | 직관적 계층 구조, 비개발 직군 친화적 |
| **Markdown 회의록** | HTML vs Markdown | 내보내기 호환성, 포맷팅 유연성, 렌더링 용이 |
