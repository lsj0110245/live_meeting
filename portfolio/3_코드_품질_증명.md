## 1. 테스트 전략

### 1.1 테스트 계층 구조

```
┌─────────────────────────────────────────────────┐
│            수동 통합 테스트 (E2E)                │  ← 전체 플로우 검증
│    녹음→STT→LLM→회의록 생성 파이프라인 검증     │
├─────────────────────────────────────────────────┤
│            API 테스트 (Integration)              │  ← FastAPI TestClient
```

```python
def test_register_and_login():
    """회원가입 → 로그인 → JWT 토큰 발급 → 인증된 API 호출"""
    # 1) 회원가입
    response = client.post("/api/auth/register", json={
        "email": "test@test.com",
        "username": "테스트",
        "password": "password123"
    })
    assert response.status_code == 201

    # 2) 로그인 → JWT 발급
    response = client.post("/api/auth/login", json={
        "email": "test@test.com",
        "password": "password123"
    })
    assert response.status_code == 200
    token = response.json()["access_token"]

    # 3) 인증된 API 호출
    response = client.get("/api/users/me",
        headers={"Authorization": f"Bearer {token}"})
    assert response.status_code == 200
    assert response.json()["email"] == "test@test.com"

def test_expired_token_returns_403():
```

## 2. 코드 아키텍처 & 유지보수성

### 2.1 레이어드 아키텍처 (Layered Architecture)

```

│              Service Layer                   │  ← 비즈니스 로직

│  security.py                                │
└─────────────────────────────────────────────┘
```

### 2.2 핵심 설계 패턴

| 패턴 | 적용 위치 | 효과 |
|------|----------|------|
| **의존성 주입 (DI)** | `deps.get_current_user()` | 모든 보호 엔드포인트에 인증 자동 적용 |
| **Fallback Chain** | LLM 파싱: `json → ast → regex → 기본 구조` | 어떤 상황에서도 빈 화면 방지 |
| **Strategy Pattern** | STT: 실시간(속도) vs 파일(정확도) 분기 | 상황별 최적 처리 전략 |
| **Observer Pattern** | 진행률 콜백 (`progress_service`) | STT/LLM 처리 진행률 실시간 업데이트 |
| **Map-Reduce** | 10,000자 초과 시 청킹 요약 | 긴 회의도 안정적으로 처리 |
| **Builder Pattern** | `RealtimeSession` 클래스 | 세션별 상태(버퍼, 히스토리, 메타데이터) 캡슐화 |

### 2.3 에러 처리 전략

```python
# 3단계 Fallback 패턴 (llm_service.py)
def _parse_llm_response(self, response: str) -> dict:
    # 1차: 정규 JSON 파싱
    try:
        return json.loads(response)
    except json.JSONDecodeError:
        pass

    # 2차: Python literal 파싱
    try:
        return ast.literal_eval(response)
    except (ValueError, SyntaxError):
        pass

    # 3차: RegEx로 JSON 블록 추출
    match = re.search(r'\{.*\}', response, re.DOTALL)
    if match:
        try:
            return json.loads(match.group())
        except:
            pass

    # 최종 Fallback: 원본 텍스트 포함 기본 구조
    return {
        "summary": {
            "purpose": "요약 생성 중 파싱 오류 발생",
            "content": response[:2000],
            ...
        }
    }
```

## 4. CI/CD 파이프라인 설계

> ⚠️ 현재 수동 배포 운영 중. 아래는 설계된 CI/CD 파이프라인 구조입니다.

### 4.1 설계된 파이프라인 (GitHub Actions)

```yaml
# .github/workflows/ci.yml (설계)
name: LiveMeeting CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # 1단계: 코드 품질 검사
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Python Lint (flake8)
        run: flake8 backend/app/ --max-line-length=120
      - name: Type Check (mypy)
        run: mypy backend/app/ --ignore-missing-imports

  # 2단계: 단위 테스트
  test:
    runs-on: ubuntu-latest
    needs: lint
    services:
      postgres:
        image: pgvector/pgvector:pg18
        env:
          POSTGRES_DB: test_db
          POSTGRES_PASSWORD: test_password
    steps:
      - uses: actions/checkout@v4
      - name: Install Dependencies
        run: pip install -r backend/requirements.txt
      - name: Run Tests
        run: pytest backend/tests/ -v --cov=backend/app

  # 3단계: Docker 이미지 빌드
  build:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - name: Build Docker Image
        run: docker-compose build --no-cache

  # 4단계: 배포 (main 브랜치만)
  deploy:
    runs-on: ubuntu-latest
    needs: build
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to Server
        run: |
          ssh ${{ secrets.SERVER_HOST }} \
            "cd /app/live_meeting && \
             docker-compose pull && \
             docker-compose up --build -d"
```

### 4.2 현재 배포 프로세스 (수동)

```bash
# 1. 코드 변경 → Git Push
git add . && git commit -m "feat: ..." && git push

# 2. 서버 접속 → Pull → 재빌드
ssh server
cd /app/live_meeting
git pull origin main
docker-compose up --build -d

# 3. 로그 확인
docker-compose logs -f backend
```

## 5. 코드 리뷰 관점의 자체 평가

### 5.1 잘된 점

| 항목 | 설명 |
|------|------|
| **에러 처리** | 3단계 Fallback 패턴으로 어떤 상황에서도 사용자에게 빈 화면을 보여주지 않음 |
| **비동기 설계** | `asyncio.to_thread()`로 GPU 연산과 이벤트 루프 완전 분리 → WebSocket 안정성 |
| **환경변수 추상화** | `.env`로 STT 모델, LLM 모델, GPU 설정 런타임 변경 가능 |
| **상태 머신** | 회의 상태 전이를 명확하게 관리 → 데이터 정합성 보장 |
| **문서화** | 6건의 트러블슈팅 사례를 Before/After 코드와 함께 상세 기록 |

### 5.2 개선이 필요한 점

| 항목 | 현재 한계 | 개선 방안 |
|------|----------|----------|
| **테스트 커버리지** | WebSocket 통합 테스트 자동화 부재 | pytest-websocket 플러그인 도입 |
| **CI/CD** | 수동 배포 | GitHub Actions 파이프라인 구축 |
| **타입 안전성** | 일부 함수에 타입 힌트 누락 | mypy strict mode 적용 |
| **코드 분리** | `recording.py` (752줄)이 과도하게 큼 | WebSocket 핸들러/세션 관리 분리 |
| **프론트엔드 상태 관리** | Vanilla JS로 전역 상태 관리 어려움 | React + TypeScript 전환 검토 |
