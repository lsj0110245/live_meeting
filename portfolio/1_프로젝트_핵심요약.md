## 🎯 이 프로젝트가 해결하는 문제

### Before (기존 방식의 문제)
| 문제 | 구체적 상황 |
|------|------------|
| ⏳ **시간 낭비** | 1시간 회의 → 회의록 작성에 30분~1시간 추가 소요 |
| 📝 **정보 누락** | 수기 기록 시 핵심 논의사항의 약 30%가 누락 |
| 🔒 **보안 우려** | ChatGPT 등 외부 API 사용 시 기업 기밀 데이터 유출 위험 |
| 🔄 **비효율적 공유** | 회의록 포맷이 다르고, 검색/관리가 불가능 |

### After (LiveMeeting의 해결)
| 해결 | 구체적 성과 |
|------|------------|
| ⚡ **2클릭 자동화** | 녹음 시작 → 녹음 종료, 두 번의 클릭만으로 구조화된 회의록 자동 생성 |
| 🎯 **AI 전사 + 요약** | Faster-Whisper STT + EXAONE LLM으로 음성→텍스트→회의록 파이프라인 완성 |
| 🏢 **완전 온프레미스** | 모든 AI 처리가 로컬 Docker 컨테이너 내부에서 수행 — 데이터 유출 제로 |
| 📊 **체계적 관리** | 폴더 분류, 검색, CSV/XLSX 내보내기로 회의록 체계적 관리 |

## 🚀 핵심 기능 시연

> 💡 아래 기능들의 실제 동작 화면은 `/portfolio/screenshots/` 디렉토리에서 확인할 수 있습니다.

### 1. 실시간 녹음 & STT (WebSocket 기반)

```
[사용자가 녹음 버튼 클릭]
    ↓  MediaRecorder API (timeslice: 500ms)
[0.5초 단위 오디오 청크 → WebSocket 전송]
    ↓  서버: 5초 버퍼 축적
[Faster-Whisper GPU 추론 (beam_size=5)]
    ↓  환각 필터링 + VAD 적용
[실시간 자막 화면 표시]
    ↓  3분마다
[LLM 중간 요약 자동 생성]
    ↓  녹음 종료
[최종 구조화된 회의록 생성]
```

**핵심 구현 포인트:**
- `asyncio.to_thread()`로 GPU 연산을 별도 스레드에서 실행 → WebSocket Ping/Pong 차단 방지
- WebM 스트리밍 헤더 캐싱으로 불완전한 오디오 청크도 안정적으로 변환
- 실시간에서는 Denoise 생략(속도 우선), 파일 업로드에서는 고품질 Denoise 적용(정확도 우선)

### 2. 파일 업로드 & 비동기 처리

```
[드래그 앤 드롭 or 클릭 업로드]
    ↓  SHA-256 해시 → 중복 파일 감지
[BackgroundTask로 비동기 STT 처리]
    ↓  30초 청킹 + 5초 오버랩 + 고품질 Denoise
[세그먼트별 LLM 전사 교정]
    ↓  진행률 0~90% 실시간 업데이트
[Map-Reduce 방식 회의록 생성 (10,000자 초과 시)]
    ↓  진행률 90~100%
[구조화된 회의록 완성 + DB 저장]
```

## 🔧 트러블슈팅 — 기술적 이슈 해결 과정

### Issue #1: WebSocket 연결 끊김 (실시간 녹음 3~5분 후)

| 항목 | 내용 |
|------|------|
| **증상** | 실시간 녹음 3~5분 경과 후 "서버 연결이 끊겨 녹음이 중단되었습니다" 알림 발생 |
| **원인** | `noisereduce` 라이브러리의 고비용 CPU 연산이 **매 청크마다** 실행 → 이벤트 루프 차단(Blocking) → WebSocket Ping/Pong 응답 불가 → 타임아웃 |
| **해결** | quality 파라미터 분기 처리 — 실시간은 Denoise 생략(속도 우선), 파일 전사는 고품질 Denoise 적용(정확도 우선) |
| **결과** | CPU 부하 대폭 감소, **장시간 실시간 녹음 안정화** |

```python
# [Before] 매 청크마다 무조건 잡음 제거 — 이벤트 루프 차단
reduced = nr.reduce_noise(y=samples, sr=16000, ...)

# [After] quality 파라미터로 분기 처리
if quality == "high":    # 파일 전사: 정확도 우선
    reduced = nr.reduce_noise(y=samples, sr=16000, ...)
else:                    # 실시간: 속도 우선, Denoise 생략
    reduced = samples
```

### Issue #2: STT 환각(Hallucination) — 무의미한 텍스트 반복 생성

| 항목 | 내용 |
|------|------|
| **증상** | "자막 박진희", "시청해 주셔서 감사합니다", "아자차카타파하" 등 무의미 텍스트 반복 |
| **원인** | ① `initial_prompt`에 테스트 예시 포함 → 모델이 그대로 따라함 ② 무음 구간에서 텍스트 강제 생성 |
| **해결** | **4단계 다층 필터링** 전략 적용 |
| **결과** | 환각 텍스트 **99% 이상 제거** |

```
① Initial Prompt 정제 → 테스트 예시 제거
② RegEx 패턴 필터링 → 환각 패턴 감지 후 스킵
③ 침묵 감지 → no_speech_prob > 0.8 자동 제거
④ 반복 억제 → no_repeat_ngram_size=3, repetition_penalty=1.3
```

### Issue #3: WebM 스트리밍 청크 — 불완전한 EBML 헤더

| 항목 | 내용 |
|------|------|
| **증상** | 실시간 녹음 시 두 번째 청크부터 pydub/ffmpeg 파싱 실패 |
| **원인** | MediaRecorder의 WebM은 **첫 번째 청크에만** EBML 헤더 포함 → 이후 청크는 독립 디코딩 불가 |
| **해결** | ① 헤더 캐싱: `RealtimeSession.header_bytes`에 첫 청크 저장 ② 헤더 병합: `header_bytes + current_data` 합성 ③ ffmpeg stdin pipe로 불완전 EBML 처리 |
| **결과** | **실시간 스트리밍 오디오의 안정적 WAV 변환 달성** |

### Issue #4: LLM 응답 JSON 파싱 실패

| 항목 | 내용 |
|------|------|
| **증상** | AI 회의록이 빈 화면으로 표시됨 |
| **원인** | LLM이 JSON 형식을 지키지 않고 Markdown 블록이나 잡담을 포함 |
| **해결** | **3단계 Fallback 파싱** 전략 |
| **결과** | **어떤 LLM 출력이든 빈 화면 없이 회의록 표시** |

```
① json.loads() → 정상 JSON 파싱
② ast.literal_eval() → Python literal 구문 파싱
③ RegEx 추출 → JSON 블록 추출 시도
④ 최종 Fallback → 원본 텍스트를 포함한 기본 구조 반환
```

### Issue #5: 회의 상태 조기 완료 (데이터 정합성)

| 항목 | 내용 |
|------|------|
| **증상** | 녹음 종료 직후 `completed` 상태이나 회의록이 아직 생성되지 않아 빈 화면 |
| **원인** | 요약 생성 **전에** 상태를 `completed`로 변경 |
| **해결** | 상태 전이 순서 변경: `recording` → `processing` → (요약 완료 후) `completed` |
| **결과** | **완전한 회의록이 생성된 후에만 "완료" 표시** |

### Issue #6: 이어서 녹음 시 파일 충돌 (Race Condition)

| 항목 | 내용 |
|------|------|
| **증상** | WebSocket 오디오 쓰기와 finalize 엔드포인트가 동시에 같은 파일 접근 → 파일 corruption |
| **원인** | 기존 파일에 단순 append → WebM 컨테이너 포맷 위반 |
| **해결** | ① 임시 파일 분리 (`_resume_{timestamp}.webm`) ② FFmpeg concat 병합 ③ Duration 헤더 복구 |
| **결과** | **이전 세션과 새 세션의 오디오가 안전하게 병합** |
