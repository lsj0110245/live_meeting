# 5장. 품질 관리 및 보안 전략

## 5.1 테스트 전략

### 5.1.1 테스트 계층 구조

| 계층 | 도구 | 대상 | 상태 |
|------|------|------|------|
| 단위 테스트 | pytest + pytest-asyncio | 서비스 함수 (STT, LLM, 유틸리티) | 구현됨 |
| API 테스트 | FastAPI TestClient | RESTful 엔드포인트 (CRUD, 인증) | 구현됨 |
| 통합 테스트 | 수동 검증 | 실시간 녹음 → STT → LLM → 회의록 전체 플로우 | 수동 수행 |
| 부하 테스트 | (계획) Locust/JMeter | 동시 WebSocket 연결, 대용량 파일 업로드 | 미구현 |

### 5.1.2 핵심 테스트 시나리오

1. **인증 플로우**: 회원가입 → 로그인 → JWT 토큰 발급 → 인증된 API 호출 → 토큰 만료 시 403 반환
2. **파일 업로드 플로우**: 파일 확장자 검증 → SHA-256 해시 중복 체크 → STT 백그라운드 처리 → DB 저장 → 자동 요약
3. **실시간 녹음 플로우**: WebSocket 연결 → 메타데이터 전송 → 오디오 청크 스트리밍 → 5초 버퍼 전사 → 3분 중간 요약 → 종료 → 최종 요약
4. **이어서 녹음 플로우**: 기존 meeting_id 전달 → segment_index 이어받기 → 별도 임시 파일 저장 → FFmpeg concat → duration 합산
5. **내보내기 플로우**: CSV(utf-8-sig) / XLSX(openpyxl 템플릿) 정상 다운로드, 한글 파일명 인코딩 확인

### 5.1.3 STT 품질 검증

| 테스트 환경 | 입력 | 기대 결과 | 실제 결과 |
|-------------|------|-----------|-----------|
| 일반 대화 (1m 거리) | 한국어 비즈니스 회의 | 문맥에 맞는 자연스러운 전사 | ✅ 정확도 양호 |
| 소음 환경 (에어컨) | 배경 잡음 + 음성 | 잡음 제거 후 음성만 인식 | ✅ noisereduce 적용 시 개선 |
| 작은 목소리 (1m+) | 속삭이는 음성 | +5dB 증폭 후 인식 | ✅ Normalize 적용 후 인식됨 |
| 전문 용어 | "LLM", "FastAPI", "Docker" | 영문 원어 그대로 전사 | ✅ large-v3-turbo로 정확히 인식 |
| 긴 회의 (10분+) | 연속 녹음 | WebSocket 연결 유지, 전사 누적 | ✅ Denoise 생략으로 안정화 |

# 6장. 주요 기술적 이슈 및 해결 전략

## 6.1 이슈 1: 실시간 녹음 중 WebSocket 연결 끊김

### 증상
- 실시간 녹음 3~5분 경과 후 "서버 연결이 끊겨 녹음이 중단되었습니다" 알림 발생

### 원인 분석
- `faster_whisper_stt_service.py`의 `_preprocess_audio()` 함수에서 **매 청크마다 noisereduce 라이브러리의 고비용 CPU 연산** 수행
- 이벤트 루프가 차단(Blocking)되어 WebSocket Ping/Pong 응답 불가 → 타임아웃 발생

### 해결 전략
```python
# [Before] 무조건 잡음 제거 수행
reduced_noise_audio = nr.reduce_noise(y=samples, sr=16000, ...)

# [After] quality 파라미터로 분기 처리
if quality == "high":    # 파일 전사용: 정확도 우선
    reduced_noise_audio = nr.reduce_noise(y=samples, sr=16000, ...)
else:                    # 실시간용: 속도 우선, Denoise 생략
    reduced_noise_audio = samples
```

### 결과
- CPU 부하 대폭 감소, WebSocket 연결 끊김 없이 장시간 실시간 녹음 가능
- 최종 회의록 품질은 녹음 종료 후 파일 업로드 재분석(Finalize)으로 유지

## 6.2 이슈 2: STT 환각(Hallucination) 텍스트 출력

### 증상
- 실시간 전사 시 "자막 박진희", "시청해 주셔서 감사합니다", "아자차카타파하" 등 무의미한 텍스트 반복 생성

### 원인 분석
- Whisper 모델의 `initial_prompt`에 테스트용 예시 문구가 포함되어 있어 모델이 이를 그대로 따라함
- 무음·잡음 구간에서 `no_speech_prob`이 높아도 텍스트를 강제 생성하는 현상

### 해결 전략
1. **Initial Prompt 정제**: 테스트 예시 제거 → "회의 녹음입니다. 자연스러운 한국어 문장으로 기록해 주세요."
2. **RegEx 필터링 추가**: 환각 패턴 감지 후 스킵
3. **침묵 감지**: `no_speech_prob > 0.8` 세그먼트 자동 제거
4. **no_repeat_ngram_size=3, repetition_penalty=1.3** 적용으로 반복 생성 억제

### 결과
- 환각 텍스트 99% 이상 제거, 실제 발화 내용만 전사

## 6.3 이슈 3: WebM 스트리밍 청크의 불완전한 헤더 문제

### 증상
- 실시간 녹음 시 두 번째 청크부터 EBML 헤더가 없어 pydub/ffmpeg가 파싱 실패

### 원인 분석
- MediaRecorder의 WebM 출력은 첫 번째 청크에만 EBML 헤더를 포함
- 이후 청크는 헤더 없이 데이터만 전송되어 독립적으로 디코딩 불가

### 해결 전략
1. **헤더 캐싱**: `RealtimeSession.header_bytes`에 첫 번째 청크(헤더 포함)를 저장
2. **헤더 병합**: 두 번째 청크부터 `header_bytes + current_data`로 합쳐서 전사
3. **skip_duration_ms**: 헤더 재결합으로 인한 중복 구간은 ffmpeg `-ss` 옵션으로 스킵
4. **ffmpeg stdin pipe**: pydub(파일 경유) 대신 `subprocess.run(cmd, input=audio_bytes)`로 직접 변환 → 불완전한 EBML도 관대하게 처리

### 결과
- 실시간 스트리밍 오디오 청크의 안정적 WAV 변환 달성

## 6.4 이슈 4: JWT 토큰 만료 시 UX 단절

### 증상
- 장시간 녹음 중 JWT 토큰이 만료되면 이후 API 호출 실패, 사용자에게 적절한 안내 없음

### 해결 전략
- `ACCESS_TOKEN_EXPIRE_MINUTES = 300` (5시간)으로 설정하여 일반 회의 시간 충분히 커버
- `common.js`의 `fetchWithAuth()`에서 403 응답 감지 시 로그인 페이지로 리다이렉트
- WebSocket 연결은 연결 수립 시점의 토큰만 검증 (연결 유지 중 만료 시에는 기존 세션 유지)

### 결과
- 토큰 만료 시 자연스러운 재인증 유도, 녹음 중 데이터는 서버에 이미 저장되어 손실 없음

## 6.5 이슈 5: 회의 상태 조기 완료 처리 (데이터 정합성)

### 증상
- 녹음 종료 직후 회의 상태가 `completed`로 변경되나, LLM 요약이 아직 생성되지 않아 빈 회의록 표시

### 원인 분석
- `recording.py`의 `final_cleanup_and_summary()`에서 요약 생성 전에 상태를 `completed`로 설정

### 해결 전략
- **상태 전이 순서 변경**: `recording` → `processing` → (요약 완료 후) `completed`
- `meeting_tasks.py`의 `process_meeting_summary()`에서 모든 처리 완료 후에만 `MeetingStatus.COMPLETED` 설정
- 에러 발생 시 `_force_complete_meeting_sync()`로 강제 완료 처리 (에러 상태에서 사용자가 무한 대기하는 것 방지)

### 결과
- 회의록이 완전히 생성된 후에만 "완료" 표시, 데이터 정합성 확보

## 6.6 이슈 6: 이어서 녹음 시 오디오 파일 충돌 (Race Condition)

### 증상
- 이어서 녹음 세션에서 기존 오디오 파일에 직접 append 시 파일 corruption 발생

### 원인 분석
- WebSocket `_write_audio_sync()`와 `finalize` 엔드포인트가 동시에 같은 파일에 접근
- 기존 파일 + 새 세션 파일을 단순 append하면 WebM 컨테이너 포맷 위반

### 해결 전략
1. **임시 파일 분리**: 이어서 녹음 세션은 `realtime_{id}_resume_{timestamp}.webm` 별도 저장
2. **FFmpeg concat**: 녹음 종료 시 `concat-resume` 엔드포인트에서 `filter_complex "[0:a][1:a]concat=n=2:v=0:a=1[a]"` 명령으로 오디오 스트림 합치기
3. **Duration 헤더 복구**: 합쳐진 파일에 대해 `_repair_audio_duration_sync()`로 FFmpeg remuxing 수행

### 결과
- 이전 세션과 새 세션의 오디오가 안전하게 병합, 타임라인 연속성 확보

## 7.3 데이터 안정성

| 위험 시나리오 | 대응 전략 | 결과 |
|---------------|-----------|------|
| WebSocket 비정상 종료 | `WebSocketDisconnect` 핸들러에서 잔여 버퍼 처리 + 백그라운드 요약 실행 | ✅ 데이터 보존 |

## 7.4 통신 구조 고도화

### 7.4.1 WebSocket 안정화 전략

1. **이벤트 루프 보호**: 모든 Blocking 연산(STT 추론, DB 쓰기, 파일 I/O)을 `asyncio.to_thread()` 또는 `asyncio.create_task()`로 비동기화
2. **연결 상태 추적**: `ConnectionManager.disconnected_clients` 집합으로 끊긴 클라이언트 감지 → 전송 시도 전 확인
3. **전송 실패 내성**: `send_json()` 실패 시 `False` 반환만 하고 예외 전파 없음 → DB 저장은 이미 완료되었으므로 데이터 무손실
4. **Pre-warm**: WebSocket 연결 즉시 STT 모델을 백그라운드에서 GPU에 로드 → 첫 전사 지연 최소화

### 7.4.2 HTTP Range Request 지원

- `main.py`의 `/media/{file_path:path}` 엔드포인트에서 HTTP 206 Partial Content 지원
- 오디오 플레이어의 시크(Seek) 기능이 정상 동작하도록 `Content-Range`, `Accept-Ranges` 헤더 구현
- ETag 기반 캐시 전략으로 변경되지 않은 파일은 304 Not Modified 반환

# 8장. 결론 및 회고

## 8.1 최종 요약

LiveMeeting 프로젝트는 **AI 기반 실시간 회의록 자동 생성 시스템**으로서 다음과 같은 핵심 가치를 구현하였다:

1. **완전한 온프레미스 AI 파이프라인**: Faster-Whisper(STT) + EXAONE 3.5(LLM)을 Docker 컨테이너 내에서 GPU 가속으로 구동하여, 기업 데이터 주권을 100% 보장하는 독립 운영 환경을 구축함.

2. **실시간 + 배치 처리 이원화**: WebSocket 기반 실시간 녹음 시에는 속도 우선(Denoise 생략, 5초 버퍼)으로, 파일 업로드 시에는 정확도 우선(고품질 Denoise, 30초 청킹, LLM 교정)으로 처리하여 상황별 최적의 UX를 제공함.

3. **견고한 에러 처리 체계**: WebSocket 연결 끊김, STT 환각, LLM 파싱 실패 등 다양한 장애 시나리오에 대한 Fallback 전략을 구현하여, 어떤 상황에서도 사용자 데이터가 보존되고 시스템이 복구되는 안정성을 확보함.

4. **체계적 데이터 관리**: Alembic을 통한 11단계 스키마 진화 관리, SHA-256 파일 해시 중복 방지, Cascade 삭제를 통한 데이터 정합성 유지.

# 부록. 소감문

## 개인 성장

LiveMeeting 프로젝트를 통해 가장 크게 성장한 부분은 **비동기 프로그래밍에 대한 깊은 이해**이다. Python의 `asyncio` 생태계, 특히 `asyncio.to_thread()`를 활용한 CPU-bound 작업의 이벤트 루프 분리를 실제 프로덕션 수준에서 적용하면서, 동시성(Concurrency)과 병렬성(Parallelism)의 차이를 체감할 수 있었다.

또한, AI 모델(Faster-Whisper, EXAONE)을 Docker 환경에서 GPU 가속과 함께 운영하면서, 모델 로딩 시간, VRAM 관리, 추론 지연 최적화 등 **AI 엔지니어링의 실무적 감각**을 쌓을 수 있었다. 특히, STT 환각(Hallucination) 문제를 `initial_prompt` 튜닝, RegEx 필터링, VAD 파라미터 조정 등 다층적 접근으로 해결한 경험은 AI 서비스의 품질 관리가 단순히 모델 성능 뿐만 아니라 전처리·후처리 파이프라인의 설계에 크게 의존한다는 교훈을 주었다.

## 아쉬운 점

프로젝트 일정상 **테스트 자동화**와 **CI/CD 파이프라인** 구축에 충분한 시간을 투자하지 못한 것이 가장 아쉽다. 특히 실시간 WebSocket 통신에 대한 자동화된 통합 테스트가 부재하여, 코드 변경 시 수동으로 전체 플로우를 검증해야 하는 비효율이 있었다.

프론트엔드를 Vanilla JavaScript로 구현한 것은 초기 개발 속도를 높였으나, 코드 규모가 커지면서(dashboard.js 44KB, recording.js 34KB) 상태 관리와 컴포넌트 재사용에 어려움을 겪었다. React나 Vue.js 같은 프레임워크를 도입했다면 유지보수성이 크게 개선되었을 것이다.
