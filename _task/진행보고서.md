# 1장. 개발 목표 및 범위 (Scope)

## 1.1 프로젝트 배경

현대 기업 환경에서 회의는 핵심 업무 활동이나, 회의 내용을 체계적으로 기록·관리하는 데 상당한 시간이 소모된다. 수기 회의록 작성은 인적 오류가 발생하기 쉽고, 핵심 논의사항이 누락되는 경우가 빈번하다. LiveMeeting은 이러한 비효율을 해결하기 위해 **AI 음성 인식(STT)** 과 **대규모 언어모델(LLM)** 을 결합하여, 회의 녹음부터 구조화된 회의록 생성까지의 전 과정을 자동화하는 온프레미스 웹 애플리케이션이다.

## 1.2 개발 목표

### 1.2.1 프론트엔드 목표
- **직관적 UI/UX**: 비개발 직군도 즉시 사용 가능한 랜딩 페이지, 대시보드, 실시간 녹음 페이지, 회의 상세 페이지 구현
- **실시간 상호작용**: WebSocket 기반 실시간 전사(Transcript) 표시, 중간 요약 자동 갱신
- **반응형 디자인**: CSS 변수 기반 다크 테마, 글래스모피즘(Glassmorphism) UI, 모바일 대응
- **파일 관리**: 폴더 기반 회의 분류, 일괄 선택·삭제·이동, 드래그 앤 드롭 파일 업로드

### 1.2.2 백엔드 목표
- **고성능 비동기 처리**: FastAPI 기반 비동기 API 서버 구축
- **온프레미스 AI 파이프라인**: 외부 클라우드 의존 없이 Faster-Whisper(STT) + EXAONE 3.5/Ollama(LLM)를 GPU 서버에서 직접 구동
- **견고한 데이터 관리**: PostgreSQL + SQLAlchemy ORM, Alembic 마이그레이션 기반 스키마 진화
- **보안**: JWT 인증, bcrypt 비밀번호 해싱, CORS 정책 제어

### 1.2.3 기술적 방향성
- **데이터 주권 확보**: 모든 음성 데이터와 회의록을 로컬 서버에 저장하여 기업 보안 정책 준수
- **확장 가능 아키텍처**: Docker Compose 기반 멀티 컨테이너 오케스트레이션으로 서비스 단위 독립 배포·확장 가능
- **AI 모델 교체 용이성**: `.env` 환경변수를 통한 STT 모델 크기, LLM 모델명, GPU 설정 등 런타임 변경 지원

## 1.3 개발 범위

# 2장. 관련 기술 분석

## 2.1 기술 스택 선정 이유

### 2.1.1 백엔드 프레임워크: FastAPI

| 선정 기준 | FastAPI | Django REST | Flask |
|-----------|---------|-------------|-------|
| 비동기 지원 | ✅ 네이티브 async/await | ⚠️ ASGI 별도 설정 | ❌ WSGI 기반 |
| WebSocket | ✅ 내장 지원 | ⚠️ Channels 추가 필요 | ⚠️ Flask-SocketIO |
| 자동 문서 | ✅ Swagger + ReDoc | ⚠️ drf-yasg | ❌ 별도 구성 |
| 성능 | ✅ Starlette 기반 고성능 | ⚠️ 보통 | ⚠️ 보통 |

**선정 사유**: 실시간 WebSocket 통신과 GPU 기반 AI 추론이 핵심 요구사항이므로, 네이티브 비동기를 지원하며 WebSocket이 내장된 FastAPI를 채택함. `asyncio.to_thread()`를 활용한 CPU-bound 작업의 이벤트 루프 분리가 용이함.

### 2.1.2 STT 엔진: Faster-Whisper (Large-v3-Turbo)

| 항목 | Faster-Whisper | OpenAI Whisper | Google STT |
|------|---------------|----------------|------------|
| 속도 | ✅ CTranslate2 기반 4~5배 빠름 | ⚠️ 기본 속도 | ✅ 빠름 |
| 메모리 | ✅ ~3.5GB VRAM | ❌ ~6GB VRAM | N/A (클라우드) |
| 데이터 주권 | ✅ 완전 로컬 | ✅ 완전 로컬 | ❌ 클라우드 전송 |
| 한국어 정확도 | ✅ large-v3-turbo로 우수 | ✅ 우수 | ✅ 우수 |

**선정 사유**: 온프레미스 운영 필수 조건 충족. CTranslate2 기반 최적화로 동일 VRAM에서 더 빠른 추론 속도 달성. `deepdml/faster-whisper-large-v3-turbo-ct2` 모델로 한국어 전문 용어(API, Docker, LLM 등)의 인식 정확도가 medium 대비 현저하게 향상됨.

### 2.1.3 LLM 서비스: EXAONE 3.5 (7.8B) via Ollama

- **한국어 특화**: LG AI Research의 EXAONE 3.5은 한국어 비즈니스 문서 생성에 최적화
- **Ollama 래퍼**: Docker 컨테이너 내에서 `ollama/ollama:latest` 이미지로 간편하게 모델 서빙
- **LangChain 통합**: `ChatOllama` + `ChatPromptTemplate` + `StrOutputParser` 체인으로 구조화된 JSON 회의록 생성
- **컨텍스트 윈도우**: `num_ctx=16384` 설정으로 긴 회의 전사 텍스트 처리 가능

### 2.1.4 데이터베이스: PostgreSQL + pgvector

- **관계형 데이터 모델링**: 사용자-회의-전사-요약 간의 명확한 1:N, 1:1 관계 표현
- **pgvector 확장**: 향후 임베딩 기반 유사 회의 검색을 위한 확장성 확보
- **Alembic 마이그레이션**: 11개 버전 이력으로 스키마 변경 추적

### 2.1.5 프론트엔드: Jinja2 + Vanilla JavaScript

- **서버 사이드 렌더링(SSR)**: FastAPI의 Jinja2 템플릿 엔진 활용으로 SEO 및 초기 로딩 속도 확보
- **SPA 유사 동작**: JavaScript를 통해 Toast 알림, 모달, 실시간 전사 업데이트 등 동적 UI 구현
- **의존성 최소화**: React/Vue 등 프레임워크 없이 순수 JS로 구현하여 빌드 파이프라인 불필요

## 2.2 시스템 설계 전략

### 2.2.1 온프레미스 AI 파이프라인 전략

```
[사용자 브라우저] ──WebSocket──▶ [FastAPI Backend] ──GPU──▶ [Faster-Whisper STT]
                                      │                        │
                                      ▼                        ▼
                              [PostgreSQL DB]          [전사 텍스트 저장]
                                      │
                                      ▼
                              [Ollama/EXAONE LLM] ──▶ [구조화된 회의록 JSON]
```

- 모든 AI 처리가 도커 네트워크 내부(`lm_network`)에서 수행됨
- 외부 API 호출 없이 완전한 독립 망 운영 가능
- GPU 자원을 STT와 LLM이 시분할로 공유 (VRAM 6GB 환경 기준)

### 2.2.2 실시간 처리 vs 배치 처리 이원화

| 구분 | 실시간 녹음 | 파일 업로드 |
|------|-------------|-------------|
| STT 전략 | 5초 버퍼링, beam_size=5, Denoise 생략 | 30초 청킹, beam_size=10, 고품질 Denoise |
| 지연 시간 | <1초 (속도 우선) | 수분 (정확도 우선) |
| 전사 교정 | 실시간 교정 비활성화 (속도) | LLM 문맥 교정 적용 |
| 요약 생성 | 3분마다 중간 요약 + 종료 시 최종 요약 | STT 완료 후 일괄 요약 |

## 3.1 전체 아키텍처

### 3.1.1 Docker Compose 서비스 구성

```yaml
services:
  db:        # PostgreSQL + pgvector (포트 15432:5432)
  backend:   # FastAPI + Faster-Whisper STT (포트 8001:8000, GPU)
  llm:       # Ollama/EXAONE 3.5 (포트 11434:11434, GPU)
```

- **db**: `pgvector/pgvector:pg18` 이미지, 영구 볼륨(`postgres_data`) 마운트
- **backend**: 커스텀 Dockerfile, Python 3.11 기반, NVIDIA GPU 1장 할당, `start.sh`로 Alembic 마이그레이션 후 uvicorn 실행
- **llm**: `ollama/ollama:latest`, 커스텀 엔트리포인트(`ollama_entrypoint.sh`)로 EXAONE 3.5 모델 자동 pull

### 3.3.2 WebSocket 프로토콜 설계

```
Client → Server (JSON):
  { "type": "metadata", "data": { "title", "meeting_type", "meeting_date", "attendees", "writer", "meeting_id" } }
  { "type": "stop_recording" }

Client → Server (Binary):
  [WebM 오디오 청크 바이너리 데이터]

Server → Client (JSON):
```

## 4.2 백엔드 고도화

### 4.2.1 STT 서비스 (`faster_whisper_stt_service.py` — 555줄)

**3-Tier STT 파이프라인:**

1. **실시간 전사 (`transcribe_realtime`)**
   - 5초 버퍼 축적 → ffmpeg stdin pipe로 WebM→WAV 변환 → Faster-Whisper GPU 추론
   - `beam_size=5`, `temperature=0`, `repetition_penalty=1.3`
   - VAD(Voice Activity Detection) 필터: `min_silence_duration_ms=800`, `threshold=0.5`
   - **환각 필터링**: RegEx 패턴으로 "자막", "감사합니다", "마이크 테스트" 등 무의미한 출력 제거
   - `no_speech_prob > 0.8` 침묵 구간 자동 스킵

2. **파일 전사 — 단일 패스 (`transcribe_file`)**
   - `beam_size=10`, `best_of=10`으로 정확도 극대화
   - `condition_on_previous_text=True`로 문맥 연속성 확보
   - 전체 파일을 한 번에 처리 (비교적 짧은 파일용)

3. **파일 전사 — 청킹 패스 (`transcribe_file_chunked`)**
   - 30초 청크 + 5초 오버랩으로 긴 파일 처리
   - `_merge_overlapping_segments()`로 오버랩 구간 중복 텍스트 자동 병합
   - 진행률 콜백을 통한 실시간 처리 상황 업데이트

**오디오 전처리 (`_preprocess_audio`):**
- **파일 모드 (high)**: noisereduce 라이브러리로 Stationary Noise 제거 → pydub Normalize → +5dB 증폭
- **실시간 모드 (fast)**: Denoise 생략, Normalize만 수행 (WebSocket 타임아웃 방지)
- **Fallback 체인**: pydub 실패 → ffmpeg pipe → 무음 WAV 반환 (오류 전파 차단)

### 4.2.2 LLM 서비스 (`llm_service.py` — 253줄)

**3가지 LLM 기능:**

1. **구조화된 회의록 생성 (`generate_summary`)**
   - LangChain `ChatPromptTemplate` + `StrOutputParser` 체인
   - JSON 출력 포맷: `{ metadata: { title_suggestion, meeting_type, attendees }, summary: { purpose, content, conclusion, action_items } }`
   - **3단계 JSON 파싱 Fallback**: `json.loads()` → `ast.literal_eval()` → RegEx 추출
   - 파싱 실패 시 원본 텍스트를 포함한 기본 구조 반환 (사용자에게 빈 화면 방지)

2. **중간 요약 (`generate_simple_summary`)**
   - 3줄 이내 간결 요약, 순수 텍스트 반환
   - 실시간 녹음 중 3분 간격으로 자동 트리거

3. **전사 텍스트 교정 (`correct_transcript`)**
   - 도메인 용어 사전(API, Docker, FastAPI, STT 등) 기반 음성 유사어 오인식 방지
   - 숫자/기호 정규화 (전화번호, 날짜 형식)
   - 사족 필터링: "교정 결과:", "수정했습니다" 등 LLM이 추가하는 불필요 텍스트 제거

### 4.2.3 실시간 녹음 WebSocket (`recording.py` — 752줄)

**핵심 아키텍처:**

- **`RealtimeSession` 클래스**: 세션별 오디오 버퍼, 전사 이력, 메타데이터, WebM 헤더 관리
- **`ConnectionManager` 클래스**: 다중 클라이언트 WebSocket 연결 관리, 전송 실패 시 `disconnected_clients` 추적
- **이벤트 루프 보호**: 모든 GPU 연산은 `asyncio.to_thread()`로 별도 스레드에서 실행하여 WebSocket Ping/Pong 차단 방지
- **이어서 녹음**: `is_resume_session` 플래그, 기존 `segment_index`와 `total_duration` 이어받기, 별도 임시 파일 저장 후 FFmpeg `concat` 병합
- **최종 정리**: `final_cleanup_and_summary()` 비동기 태스크에서 잔여 버퍼 처리 → duration 업데이트 → LLM 요약 생성 → meeting 상태 `COMPLETED` 전환

### 4.2.4 파일 업로드 처리 (`upload.py` — 384줄)

- **SHA-256 중복 방지**: 업로드 시 파일 해시 계산, 기존 동일 파일 발견 시 재사용
- **BackgroundTask 기반 비동기 STT**: `process_audio_file()` 함수를 백그라운드에서 실행 → STT → 세그먼트별 LLM 교정 → DB 저장 → 자동 요약 생성
- **진행률 관리**: `progress_service`를 통한 0~100% 진행률 업데이트 (STT 0~90%, 요약 90~100%)
- **finalize 엔드포인트**: 실시간 녹음 종료 후 완성된 WebM 파일 수신, FFmpeg Duration 헤더 복구

### 4.2.5 내보내기 서비스 (`export.py` — 236줄)

- **CSV**: `pandas` + `utf-8-sig` BOM으로 한글 인코딩 보장
- **XLSX**: `openpyxl`로 `source.xlsx` 템플릿 기반 생성, 셀 스타일(줄바꿈, 정렬, 열 너비) 자동 설정
- **AI 요약 파싱**: Markdown 섹션별(📅📌✅📝) RegEx 파싱 → 엑셀 셀 분배
- **엑셀 수식 오류 방지**: 대시(`-`) 불렛을 `●`로 치환 (엑셀이 `-`를 수식으로 해석하는 문제 방지)

### 4.2.6 Map-Reduce 요약 전략 (`meeting_tasks.py` — 251줄)

- **안전 제한**: 전사 텍스트 10,000자 초과 시 Map-Reduce 전략 자동 전환
- **Map**: 3,000자 청크 + 500자 오버랩으로 분할 → 각 청크별 `generate_simple_summary()` 호출
- **Reduce**: 부분 요약들을 통합하여 `generate_summary()`로 최종 구조화된 회의록 생성
- **진행률 시각화**: `slow_climb()` 비동기 태스크로 2초마다 1%씩 진행률 상승 (사용자 체감 UX 향상)
