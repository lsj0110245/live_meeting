## 2. 기술 선택의 이유 — "왜 이 기술을 선택했는가?"

### 2.1 FastAPI — "비동기가 필수였다"

**고민:**
> 프로젝트 초기에 Django REST Framework와 FastAPI 사이에서 고민했다. Django는 생태계가 풍부하고 Admin 패널이 기본 제공되어 빠른 프로토타이핑에 유리하지만, 이 프로젝트의 기술적 요구사항은 달랐다.

**결정 근거:**
1. **WebSocket 내장**: 실시간 녹음 → STT 전사 → 클라이언트 실시간 전송이 핵심 기능. Django Channels는 별도 설정이 필요하지만, FastAPI는 네이티브로 WebSocket을 지원한다.
2. **비동기 I/O**: GPU에서 STT 추론이 실행되는 동안 WebSocket Ping/Pong이 중단되면 안 된다. `async/await` 네이티브 지원과 `asyncio.to_thread()`가 결정적이었다.
3. **자동 API 문서화**: Swagger + ReDoc 자동 생성. 18개 엔드포인트의 문서를 수동으로 관리하는 것보다 효율적이다.

**돌이켜보면:**
FastAPI 선택은 100% 올바른 결정이었다. 특히 `asyncio.to_thread()`로 GPU 연산을 별도 스레드에서 실행하여 이벤트 루프를 보호하는 패턴은 이 프로젝트의 안정성을 결정짓는 핵심 요소가 되었다.

---

### 2.2 Faster-Whisper — "속도와 정확도 사이의 균형점"

**고민:**
> STT 엔진 선택에서 3가지 옵션을 검토했다:
> 1. OpenAI Whisper (원본) — 정확하지만 느리고 VRAM 6GB 소모
> 2. Google Cloud STT — 빠르고 정확하지만 클라우드 의존 → 데이터 주권 문제
> 3. Faster-Whisper — CTranslate2 기반 최적화, 로컬 구동

**전환 과정:**
초기에는 `medium` 모델로 시작했으나, "LLM", "FastAPI", "Docker" 같은 전문 용어 인식 정확도가 낮았다. `.env` 설정을 통해 `deepdml/faster-whisper-large-v3-turbo-ct2`로 업그레이드했고, 이 과정에서 **환경변수를 통한 모델 교체 추상화**가 큰 효과를 발휘했다.

```
# 모델 전환이 .env 한 줄 변경으로 가능
STT_MODEL_SIZE=deepdml/faster-whisper-large-v3-turbo-ct2
```

**배운 점:**
- AI 모델 선택은 단순히 벤치마크 성능이 아니라, **운영 환경(VRAM, 속도, 데이터 주권)** 을 종합적으로 고려해야 한다.
- 모델 교체 가능성을 설계 단계에서 고려하면 이후 유연성이 극적으로 향상된다.

### 2.3 Vanilla JS — "빠른 시작, 나중의 대가"

**고민:**
> React나 Vue.js를 사용할지, Vanilla JavaScript로 갈지 선택해야 했다.

**당시 판단:**
- 프로젝트 일정이 5주로 타이트하다.
- FastAPI의 Jinja2 SSR을 활용하면 빌드 파이프라인이 불필요하다.
- 별도의 프론트엔드 서버 없이 백엔드가 직접 서빙한다.
- 핵심 기능 구현에 집중하는 것이 우선이다.

**결과적 한계:**
```
dashboard.js  →  44,153 bytes (약 1,200줄)
recording.js  →  34,759 bytes (약 900줄)
meeting.js    →  23,036 bytes (약 600줄)
```

파일 크기가 커지면서 **상태 관리의 어려움**이 체감되었다. 전역 변수로 상태를 관리하다 보니 예상치 못한 사이드 이펙트가 발생했고, 컴포넌트 재사용도 어려웠다.

**다시 만든다면:**
React + TypeScript 조합을 선택할 것이다. 초기 설정 비용이 들지만, 코드 규모가 커질 때의 유지보수성이 현저하게 다르다. 특히 WebSocket 상태 관리에 React Hooks(`useEffect`, `useRef`)가 Vanilla JS의 전역 변수보다 훨씬 안전하다.

### 2.4 온프레미스 — "보안이 서비스의 가치"

**고민:**
> ChatGPT API를 사용하면 회의록 품질이 훨씬 좋을 것이다. 하지만...

**결정:**
기업 회의 내용은 **기밀 정보**를 포함할 수 있다. "외부로 데이터가 전송되지 않는다"는 것은 단순한 기술적 제약이 아니라, **서비스의 핵심 가치 제안(Value Proposition)**이다.

Ollama + EXAONE 3.5를 Docker 컨테이너에서 구동하면서, 인터넷 연결 없는 폐쇄망 환경에서도 완벽하게 동작하는 시스템을 구축했다.

**배운 점:**
- 온프레미스 AI는 모델 품질 외에도 **모델 로딩 시간, VRAM 관리, 추론 지연** 같은 운영 이슈를 직접 해결해야 한다.
- 클라우드 API 호출과 로컬 GPU 추론은 개발 난이도가 완전히 다르다.

## 3. 가장 어려웠던 순간 — "WebSocket이 끊기는 이유를 찾는 데 이틀"

### 3.1 문제 상황

실시간 녹음 기능을 구현한 후, 테스트에서 3~5분만 녹음하면 WebSocket 연결이 끊겼다. 처음에는 네트워크 문제, 브라우저 제한, 서버 메모리 등 여러 가설을 세웠다.

### 3.2 디버깅 과정

```
가설 1: 브라우저의 WebSocket 연결 제한 → 테스트 결과 아님
가설 2: 서버 메모리 부족 → 모니터링 결과 정상
가설 3: 오디오 청크 크기 문제 → 조절해도 동일하게 끊김
가설 4: 이벤트 루프 차단 → ✅ asyncio 로그로 확인!
```

`noisereduce` 라이브러리의 Spectral Gating 알고리즘이 **CPU-bound 연산**이었고, 이것이 `asyncio` 이벤트 루프를 차단하고 있었다. 결과적으로 WebSocket의 Ping/Pong 응답이 타임아웃되어 연결이 끊겼다.

### 3.3 해결

`quality` 파라미터를 도입하여 실시간 모드에서는 Denoise를 생략하고, 파일 전사 모드에서만 고품질 Denoise를 적용했다.

### 3.4 교훈

> "비동기 프로그래밍에서 CPU-bound 작업은 이벤트 루프의 적이다."

이 경험 이후 모든 GPU 연산과 파일 I/O를 `asyncio.to_thread()`로 감싸는 습관이 생겼다. 동시성(Concurrency)과 병렬성(Parallelism)의 차이를 코드가 아닌 **장애 상황**에서 체감한 순간이었다.

## 4. 가장 성장한 부분

### 4.1 비동기 프로그래밍에 대한 깊은 이해

| Before | After |
|--------|-------|
| `async/await`를 HTTP 라우터에만 사용 | GPU 연산, DB 쓰기, 파일 I/O 전부 비동기 분리 |
| 동기 함수 내에서 블로킹 호출 | `asyncio.to_thread()`, `asyncio.create_task()` 적극 활용 |
| 이벤트 루프 개념이 추상적 | WebSocket Ping/Pong 차단 문제를 직접 디버깅하며 체화 |

### 4.2 AI 엔지니어링 실무 감각

| 영역 | 체득한 지식 |
|------|------------|
| **STT 품질 관리** | 모델 성능만이 아닌 전처리(Denoise, Normalize)/후처리(환각 필터, 전사 교정) 파이프라인의 중요성 |
| **LLM 프롬프트 엔지니어링** | JSON 출력 강제, 페르소나 설정, Fallback 파싱 전략 |
| **GPU 자원 관리** | VRAM 6GB 환경에서 STT + LLM 시분할 운영, 모델 로딩 시간 최적화 |
| **Map-Reduce 패턴** | 10,000자 초과 텍스트의 안정적 요약을 위한 청킹 전략 |

### 4.3 문제 해결 방법론

이 프로젝트에서 6건의 기술적 이슈를 해결하면서 자연스럽게 체계적인 문제 해결 방법론이 형성되었다:

```
1. 증상 정확히 기록 (재현 조건 포함)
2. 가설 목록 작성 (최소 3개)
3. 가설별 검증 실험 설계
4. 로그/모니터링으로 원인 특정
5. Before/After 코드로 해결책 문서화
6. 재발 방지를 위한 아키텍처 개선
```

## 5. 아쉬운 점과 개선 방향 — "다시 만든다면?"

### 5.1 테스트 자동화 부재

**아쉬운 점:**
WebSocket 실시간 녹음 플로우에 대한 자동화된 통합 테스트가 없다. 코드 변경 시마다 수동으로 녹음 → STT → 회의록 전체 플로우를 검증해야 했다.

**다시 만든다면:**
```python
# pytest-websocket을 활용한 자동화 테스트
@pytest.mark.asyncio
async def test_realtime_recording_flow():
    async with websocket_client("/api/recording/ws/test") as ws:
        # 1. 메타데이터 전송
        await ws.send_json({"type": "metadata", "data": {...}})
        response = await ws.receive_json()
        assert response["type"] == "meeting_created"

        # 2. 오디오 청크 전송
        audio_chunk = load_test_audio("sample.webm")
        await ws.send_bytes(audio_chunk)

        # 3. 전사 결과 수신
        response = await ws.receive_json()
        assert response["type"] == "transcript"
        assert len(response["text"]) > 0
```

### 5.2 프론트엔드 아키텍처

**아쉬운 점:**
Vanilla JS의 전역 상태 관리가 코드 규모 증가에 따라 유지보수 비용을 높였다.

**다시 만든다면:**
- **React + TypeScript**: 컴포넌트 기반 아키텍처로 재사용성 확보
- **Zustand/Recoil**: 경량 상태 관리 라이브러리 도입
- **React Query (TanStack Query)**: API 캐싱 + 실시간 업데이트 관리

### 5.3 CI/CD 파이프라인

**아쉬운 점:**
수동 배포로 인한 비효율. Git Push → SSH 접속 → Docker 재빌드의 반복.

**다시 만든다면:**
- GitHub Actions로 자동 테스트 + Docker 이미지 빌드
- Docker Hub 자동 Push + Webhook 기반 서버 배포

### 5.4 화자 분리 (Speaker Diarization)

**아쉬운 점:**
현재 모든 전사 세그먼트의 `speaker`가 `"Unknown"`으로 고정되어 있다. 누가 무엇을 말했는지 구분할 수 없다.

**다시 만든다면:**
- `pyannote-audio` 또는 NVIDIA NeMo 기반 화자 분리 통합
- 화자별 발화 구간 매핑 → 회의록에 "김팀장: ...", "이대리: ..." 형태로 표시

## 7. 마무리

> LiveMeeting은 단순한 "회의록 자동 생성 도구"가 아니라, **온프레미스 환경에서 AI를 실제 서비스로 구현하는 전 과정을 경험한 프로젝트**입니다.
>
> 가장 값진 경험은 "작동하는 코드"를 넘어서 **"장애 상황에서도 안정적으로 동작하는 시스템"**을 만들기 위해 고민한 과정입니다. WebSocket 연결 끊김, STT 환각, LLM 파싱 실패 등 실제 운영에서 발생하는 문제들을 직접 분석하고 해결하면서, 단순한 기능 구현을 넘어 **프로덕션 품질의 소프트웨어**가 무엇인지 체감할 수 있었습니다.
>
> 이 프로젝트를 통해 배운 가장 중요한 교훈은:
> 1. **기술 선택은 트레이드오프**: 정답은 없고, 프로젝트의 맥락(일정, 요구사항, 운영 환경)에 맞는 결정이 중요하다.
> 2. **에러 처리가 핵심 역량**: Happy Path만 구현하면 50%. 나머지 50%는 장애 시나리오 대응이다.
> 3. **문서화는 미래의 나를 위한 것**: 트러블슈팅을 문서화해두면, 같은 실수를 반복하지 않는다.
